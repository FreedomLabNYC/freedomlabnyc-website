Skill Creator for AI Agents

Skill AI Agents

A meta-skill: teach your AI agent how to create, edit, and audit skills for itself and others.

Sharing a skill? Post the finished SKILL.md or .skill file in the Freedom Lab Telegram group so other people can inspect, adapt, and try it with their own agents.

⚠️ Before installing or using any community skill, review its full contents — including scripts — to ensure it meets your security and quality standards. Users take responsibility for skills they choose to use.

Install This Skill

Hermes Agent: install from the public skills registry:

hermes skills inspect FreedomLabNYC/skills/skill-creator
hermes skills install FreedomLabNYC/skills/skill-creator --yes

Other agents: download the portable .skill package below, or point your agent at this page and ask it to inspect the skill contents before using it.

Download skill-creator.skill

What This Skill Does

This is a skill that teaches your AI agent how to create other skills: understand the workflow, write a clean SKILL.md, add only useful resources, polish the result, and package it so different agents can inspect and reuse it.

Think of it as the "teach a man to fish" skill. The goal is not publishing to a central website; the goal is helping people share useful, portable skills with each other — especially in the Freedom Lab Telegram group.

When to Use It

  • Creating a new skill from scratch
  • Improving, reviewing, or auditing an existing skill
  • Restructuring a skill directory
  • Validating a skill against the AgentSkills spec
  • Preparing a skill to share in Telegram or another community channel
  • Making a skill readable by Hermes, Codex-style agents, Claude-style agents, and other tool-using agents

About Skills

Skills are modular, self-contained packages that extend your AI agent's capabilities by providing specialized knowledge, workflows, and tools. They transform a general-purpose agent into a specialized one equipped with procedural knowledge that no model can fully possess.

What Skills Provide

  1. Specialized workflows — Multi-step procedures for specific domains
  2. Tool integrations — Instructions for working with specific file formats or APIs
  3. Domain expertise — Company-specific knowledge, schemas, business logic
  4. Bundled resources — Scripts, references, and assets for complex and repetitive tasks

Core Principles

Concise is Key

Keep SKILL.md lean. Include only procedural details, examples, and tool-specific knowledge that help the agent perform the workflow better than it could from general reasoning alone.

Default assumption: the agent is already very smart. Only add context it doesn't already have. Challenge each piece of information: "Does the agent really need this explanation?" and "Does this paragraph justify its token cost?"

Set Appropriate Degrees of Freedom

Match the level of specificity to the task's fragility and variability:

  • High freedom (text-based instructions): Multiple approaches are valid, decisions depend on context
  • Medium freedom (pseudocode or scripts with parameters): A preferred pattern exists, some variation acceptable
  • Low freedom (specific scripts, few parameters): Operations are fragile, consistency is critical

Anatomy of a Skill

Every skill consists of a required SKILL.md file and optional bundled resources:

skill-name/
├── SKILL.md (required)
│   ├── YAML frontmatter metadata (required)
│   │   ├── name: (required)
│   │   └── description: (required)
│   └── Markdown instructions (required)
└── Bundled Resources (optional)
    ├── scripts/          - Executable code (Python/Bash/etc.)
    ├── references/       - Documentation for context loading
    └── assets/           - Files used in output (templates, icons, fonts)

SKILL.md (required)

  • Frontmatter (YAML): Contains name and description fields. These determine when the skill gets triggered — be clear and comprehensive.
  • Body (Markdown): Instructions and guidance. Only loaded after the skill triggers.

Scripts (scripts/)

Executable code for tasks that require deterministic reliability or are repeatedly rewritten. Token efficient and may be executed without loading into context.

References (references/)

Documentation loaded as needed into context. Keeps SKILL.md lean. For large files (>10k words), include grep search patterns in SKILL.md.

Assets (assets/)

Files used in output but not loaded into context — templates, images, icons, boilerplate code, fonts.

Skill Creation Process

Step 1: Understand with Concrete Examples

Clearly understand how the skill will be used. Ask about functionality, usage patterns, and triggers. Start with the most important questions and follow up as needed.

Step 2: Plan Reusable Contents

Analyze each example to identify what scripts, references, and assets would be helpful. Turn concrete examples into a list of reusable resources.

Step 3: Initialize the Skill

Run the initialization script to generate a template skill directory:

scripts/init_skill.py <skill-name> --path <output-directory> \
    [--resources scripts,references,assets] [--examples]

Step 4: Edit the Skill

Implement the reusable resources and update SKILL.md. Remember: the skill is being created for another agent instance to use. Include information that would be beneficial and non-obvious.

Test any added scripts by actually running them. Use imperative/infinitive form in writing.

Step 5: Package the Skill

scripts/package_skill.py <path/to/skill-folder>

The script validates the skill (frontmatter, naming, structure, description quality) and creates a distributable .skill file. Fix any validation errors and re-run if needed.

Step 6: Polish for Sharing

Before sharing, make the skill easy for another person and another agent to trust: remove private paths or secrets, replace local assumptions with portable alternatives, keep setup steps explicit, and include a short note explaining what was tested.

Step 7: Share and Iterate

Share either the SKILL.md text or the packaged .skill file in the Freedom Lab Telegram group. Ask people to inspect it before use, try it with their own agent, and report what was confusing. Use that feedback to simplify the skill and test again.

Make It Accessible to Different Agents

  • Plain Markdown first: make SKILL.md useful even if the receiving agent cannot install .skill packages.
  • Portable paths: avoid hardcoded local paths, private usernames, and machine-specific assumptions. Give OS-neutral alternatives when possible.
  • Minimal dependencies: list required commands or API keys clearly, and include graceful fallbacks where possible.
  • Agent-neutral language: write for “the agent,” not only Codex, Claude, Hermes, or one user’s setup.
  • Safe sharing: never include secrets, private data, raw credentials, session logs, or unpublished internal details.
  • Two share formats: post the SKILL.md for quick review, and attach the .skill archive when bundled scripts/references/assets matter.

Progressive Disclosure

Skills use a three-level loading system:

  1. Metadata (name + description) — Always in context (~100 words)
  2. SKILL.md body — When skill triggers (<5k words)
  3. Bundled resources — As needed (unlimited since scripts can execute without reading into context)

Keep SKILL.md under 500 lines. Split content into separate files when approaching this limit, and reference them clearly from SKILL.md.

Share in the Freedom Lab Telegram Group

We are not using this workflow to publish skills directly to the website. Instead, create and polish the skill, then share it in Telegram for community review.

  1. Post a short summary: what the skill does, who it is for, and what you tested.
  2. Paste the SKILL.md contents, or attach the packaged .skill file if it has bundled resources.
  3. Tell people what to inspect before using it, especially scripts and setup steps.
  4. Invite feedback from people using different agents, then update the skill so it becomes more portable.
